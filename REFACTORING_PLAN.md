# ぴよ氏の冒険 — リファクタリング計画書

**作成日:** 2026-06-11（Ver.1.271時点）
**目的:** 長年の機能追加で蓄積した重複・無駄を段階的に解消し、今後の開発速度とバグ耐性を上げる

---

## この計画の読み方（サマリー）

ゲームの見た目・動作は**一切変えません**。中身のコードだけを整理します。
全7フェーズに分け、**1フェーズ＝1つのPR（マージ単位）**で進めます。各フェーズ完了ごとに実機でプレイテストし、問題があればそのPRだけを取り消せる構造にします。

| フェーズ | 内容 | リスク | 効果 |
|---------|------|--------|------|
| 0 | 安全網の構築（テスト手順書） | なし | 以降の全フェーズの保険 |
| 1 | デッドコード掃除・マジックナンバー定数化 | 低 | 小 |
| 2 | 重複コードの関数化（バグの温床を潰す） | 低 | **大** |
| 3 | 画面管理の統一 | 中 | 大 |
| 4 | ショップ2系統の統合 | 中〜高 | 大 |
| 5 | 敵・エフェクト処理の整理 | 中 | 中 |
| 6 | インラインCSSの整理 | 低〜中 | 中 |
| 7 | ファイル分割（**要相談**・任意） | 中 | 中 |

**推定効果:** 9,087行 → 約7,000行（20〜25%削減）。ただし本質は行数ではなく「同じ修正を3箇所にする必要がなくなる」「チェック漏れ系バグが構造的に再発しなくなる」こと。

## 共通ルール（全フェーズ適用）

1. **挙動変更ゼロ** — 見た目・操作感・数値は一切変えない。純粋なリファクタリング
2. **1フェーズ＝1PR** — マージして実機確認してから次のフェーズへ。問題発生時はそのPRのみRevert
3. **バージョン管理** — フェーズごとに +0.001、sw.jsのCACHE_NAME同時更新（CLAUDE.mdルール）
4. **フェーズ完了条件** — TESTING.md（フェーズ0で作成）の全項目をユーザーが実機でプレイ確認
5. **変更禁止事項の厳守** — デッドゾーン37%等、CLAUDE.mdの禁止事項はリファクタリング中も不可侵
6. **ビルドツール導入なし** — npm/バンドラ等は使わない。GitHubのWebアップロードだけで完結する現在の運用を維持

## フェーズ2: 重複コードの関数化（最重要フェーズ）

**目的:** 「同じコードが複数箇所にあるせいで、修正漏れがバグになる」構造を潰す。過去バグの再発防止に直結。

### やること
1. **`bindTapButton(el, handler)` ヘルパー新設**
   - 17箇所の `xxxTouchFired` フラグ＋touchend＋clickの二重発火対策パターンを1関数に集約
   - 対象: pauseBtn, soundToggle, submitBtn, skipBtn, retry, toTitle, adRevive, ranking, settings, shopItem, tshopItem, cfYes/No, tcfYes/No, depBtn, ssClose
2. **`isPlayerProtected()` ヘルパー新設**
   - `gameState.isInvincible || gameState.puShield > 0` を1関数に
   - 適用箇所: 地上敵、飛行敵、ボス体当たり、卵、閃光（5箇所）
   - **今後新しい攻撃を追加するときはこの関数を呼ぶだけでチェック漏れが起きない**
3. **`setKeeperTextFor(elementId, key, replacements)` への統合**
   - setKeeperText / setTshopKeeperText のほぼ同一実装を1つに
4. **DOM要素キャッシュの徹底**
   - 既存の `uiElements` パターンを拡張し、初期化時に全要素を1回だけ取得
   - 168回の getElementById を削減（特に毎フレーム実行されるupdateUI周辺を優先）
5. **SoundManagerの整理**
   - 各SE関数の手書きオシレーター設定を、既存の `_osc()` ヘルパーベースに書き直し
   - mp3 SE再生の共通化（currentTime=0 → play() パターン）

### 完了条件
- TESTING.md全項目パス。特にシールドでボスの全攻撃が無効化されること、全ボタンが1回タップで1回だけ反応すること

## フェーズ3: 画面管理の統一

**目的:** 34個のshow/hide/close関数と、popstateハンドラの巨大if連鎖を、登録制の画面マネージャに統一。

### やること
1. **ScreenManagerの新設**
   ```js
   Screens.register('settings', { el: '#settingsScreen', onShow: updateSettingsUI });
   Screens.show('settings');   // pushState・遷移クールダウン・hidden切替を内包
   Screens.back();             // history.back()経由で閉じる
   ```
2. **対象画面（8つ）:** settings / guide / ranking / nameInput / gameOver / pause / stageShop / titleShop
3. **popstate（Android戻るボタン）処理の一元化** — 現在の巨大if連鎖をScreenManagerのスタック管理に置き換え
4. **遷移クールダウン（isInTransitionCooldown/markScreenTransition）をScreenManager内部に移動**

### 注意点
- 戻るボタンの挙動（ショップ中は退店確認、ゲーム中はポーズ等）の例外処理を正確に移植
- モーダル（gameModal）のPromiseベース実装は別系統として維持

### 完了条件
- TESTING.md全項目パス＋Android実機で戻るボタンの全パターン確認

## フェーズ4: ショップシステムの統合

**目的:** ステージショップとタイトルショップの並行実装（メニュー描画・確認ダイアログ・カーソル処理・Yes/No処理が二重）を共通エンジン化。

### やること
1. **共通ShopUIエンジンの抽出**
   - メニュー項目レンダリング（renderShopMenuItem系）
   - 2タップ確認フロー（1回目ハイライト→2回目確定）
   - はい/いいえ確認ボックス（shopConfirmBox / tshopConfirmBox）
   - 店員セリフ表示
2. **各ショップは「設定オブジェクト」になる**
   - 商品リスト、通貨（score/savings）、店員テキストの口調キー、背景画像セットをデータとして分離
   - **口調取り違えバグの構造的防止:** 店員のセリフキーがショップ定義に紐付くため、誤参照がそもそも起きない
3. **updateStageShopUI 25回呼び出しの削減** — 購入/売却/貯金の各アクション末尾で自動再描画する構造に

### 注意点
- 金額計算（購入・売却50%・貯金50%・広告ボーナス）は1円もズレてはいけない。フェーズ0のチェックリストに金額検証項目を追加
- 課金Stub（storeScreen）は対象外（Capacitor導入時に作り直すため）
